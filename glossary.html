Glossary of Terms

A guide to terms used throughout OpenCounter.

      Applications

    • "Application"

      An 'application' refers to any OpenCounter session initiated by a member of the public. In OpenCounter, an 'application' refers to allinformation and responses an applicant needs to provide to accomplish their goal--even if that may require securing multiple Existing Permits

    • "Applicant" or "Permit Applicant"

      An applicant or permit applicant refers to any member of the public that has initiated an OpenCounter session. Oftentimes, applicants are only using OpenCounter to learn more information--not all applicants will submit an application.

    • "Help Requests", "Notes" and the "Admin Papertrail"

      Help Requests are submitted by applicants when they are in need of assistance during an OpenCounter session. Help Requests are routed to all of the Admin Users in a particular Departmentdepending on which Page the applicant needs help with--that way, people are connected with the staff that can help, and staff doesn't have worry about routing questions to the right person. Notes are recorded by Admin Users to document communication with applicants--whether that was a response to a Help Request, or a proactive message sent to people with applicants in-progress. The Admin Papertrail is a log of all notes add by all Admin Users, which can be reviewed on an application by application basis.

    • "Forms" and "Downloads"

      OpenCounter Forms reflect your organization's Existing Permit Forms. When OpenCounter was first configured in your community, all Existing Permit Forms were collected, and broken down into their component fields. To create a form in OpenCounter, simply group together the fields that compose each existing permit form. When an application is submitted, your staff will be able to download exactly the information needed for each permit process in the "Download" section.

      Zoning

    • "Zoning District"

      The zoning districts in OpenCounter were configured to reflect your community's zoning ordinance. They are geographic districts that determine what kind of land use is appropriate in a given location. In OpenCounter, zoning districts provide a zoning clearance for each use code. the OpenCounter system was configured to match the existing zoning ordinance precisely. Thus, changes to the zoning districts in the OpenCounter system should probably only follow changes to the legally binding zoning ordinance itself.

    • Use Code

      Use codes are used to classify different prospective business types. They were configured to reflect your community's zoning ordinance. In combination with a prospective location or address, the use code associated with an application results in a zoning clerance. Zoning clearances can also be described as a use code "permission".

    • Zoning Clearance (or Permissions)

      Zoning clearances are the evaluation of prospective businesses at prospective locations. Zoning districts have different allowances for different use codes. In OpenCounter, there are three primary clearances: Principally Permitted, Conditionally Permitted and Prohibited.

    • "Principally Permitted"

      Principally permitted applications are those prospective businesses whose use code is permitted at the proposed address without adjustment by the applicant.

    • "Conditionally Permitted"

      Conditionally permitted applications are those prospective businesses whose use code is only permitted at the proposed address if certain conditions are met. These conditions vary considerably across jurisdictions. You can communicate what conditions must be met--and how the applicant should go about meeting them--as your organization sees fit.

    • "Prohibited"

      Prohibted applications are those prospective businesses whose use code is not permitted at the proposed location. Applicants seeking to open a business whose use code is not permitted in the desired location are good candidates for location assistance.

      Application Flows

    • "Sections"

      Sections are collections of Pages that contain similar information. Their primary purpose is to make it easier for applicants to understand what kind of information they will need to provide throughout the application process. Sections can be re-arranged at your discretion.

    • "Pages"

      Pages are just what they sound like: single web-pages. Each page can be used to communicate information to applicants, or solicit information from them with fields. Pages are grouped into Sections, and can be arranged within each Section at your discretion.

      A key feature of OpenCounter is the ability to show or hide Pages using Visibility Rules

    • "Fields"

      Fields are requests for information from applicants. Each Field is a request for a single piece of infomration. When OpenCounter was configured in your community, your Existing Permit Forms were broken down into the constiuent fields. You can add, remove or edit Fields at your discretion. Fields can be grouped into Pages, as well as Forms.
